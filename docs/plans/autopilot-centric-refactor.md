# Autopilot-Centric Blueprint Management

**Status:** Draft
**Date:** 2026-03-06
**Author:** ClawUI Architecture Team

## Motivation

Today the autopilot loop and manual AI operations (enrich, split, smart-deps, reevaluate, generate) are two separate code paths:

1. **Autopilot loop** (`autopilot.ts:runAutopilotLoop`) — iterative observe/decide/execute cycle. The AI picks a tool from a hardcoded palette, and `executeDecision` dispatches it.
2. **Manual AI operations** (`plan-operations.ts`) — route handlers in `plan-routes.ts` call `enrichNodeInternal`, `splitNodeInternal`, etc. directly. These spawn one-shot Claude sessions with bespoke prompts.

This dual-path architecture creates problems:

- **No user-to-autopilot communication channel.** Once autopilot is running, users can only pause it — they can't steer it with messages like "skip nodes 3-5" or "focus on the API layer first."
- **AI operations are duplicated.** `enrich_node` in the autopilot palette calls `enrichNodeInternal` (which spawns a sub-agent with a hardcoded prompt), while the route handler does the same thing. The autopilot could just read the node, decide what to change, and call `update_node` directly — no sub-agent needed.
- **State snapshot is monolithic.** `buildStateSnapshot` serializes the entire blueprint into the prompt every iteration. For large blueprints this wastes context tokens. The autopilot should be able to self-serve specific pieces of context on demand.

This design doc describes a refactor that:
1. Adds a **message queue** so users can communicate with the running autopilot
2. Provides **lightweight read-only endpoints** the autopilot can call to self-serve context
3. **Removes sub-agent tools** from the autopilot palette in favor of direct CRUD + read tools
4. **Routes manual AI operations through the message queue** when in autopilot/FSD mode
5. Adds a **chat UI** to the blueprint detail page

## Section 1 — Message Queue Schema

### Table: `autopilot_messages`

```sql
CREATE TABLE IF NOT EXISTS autopilot_messages (
  id            TEXT PRIMARY KEY,
  blueprint_id  TEXT NOT NULL REFERENCES blueprints(id) ON DELETE CASCADE,
  role          TEXT NOT NULL,          -- "user" | "system"
  content       TEXT NOT NULL,
  acknowledged  INTEGER DEFAULT 0,
  created_at    TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_autopilot_messages_unack
  ON autopilot_messages(blueprint_id, acknowledged, created_at);
```

**Column semantics:**

| Column | Description |
|---|---|
| `id` | UUID, primary key |
| `blueprint_id` | FK to `blueprints.id` |
| `role` | `"user"` — human-initiated messages (typed in chat UI, or generated by route handlers redirecting AI ops). `"system"` — autopilot acknowledgments, status updates. |
| `content` | Free-text message body. For redirected AI ops, uses structured prefixes like `"[enrich] Node {nodeId}: {title}"` |
| `acknowledged` | `0` = unread by autopilot, `1` = autopilot has consumed it |
| `created_at` | ISO 8601 timestamp |

### CRUD helpers in `plan-db.ts`

```typescript
// Create a new message
export function createAutopilotMessage(
  blueprintId: string,
  role: "user" | "system",
  content: string,
): AutopilotMessage

// Get unacknowledged messages for a blueprint (ordered by created_at ASC)
export function getUnacknowledgedMessages(
  blueprintId: string,
): AutopilotMessage[]

// Mark a single message as acknowledged
export function acknowledgeMessage(messageId: string): boolean

// Paginated message history (newest first)
export function getMessageHistory(
  blueprintId: string,
  limit?: number,   // default 50
  offset?: number,  // default 0
): AutopilotMessage[]
```

**Interface:**

```typescript
export interface AutopilotMessage {
  id: string;
  blueprintId: string;
  role: "user" | "system";
  content: string;
  acknowledged: boolean;
  createdAt: string;
}
```

### Migration strategy

Incremental migration in `initPlanTables()` — check for table existence, create if missing. Follows the existing pattern (see `convene_sessions`, `blueprint_insights` migrations at `plan-db.ts:477-530`).

```typescript
const msgTables = db.prepare(
  "SELECT name FROM sqlite_master WHERE type='table' AND name='autopilot_messages'"
).all();
if (msgTables.length === 0) {
  db.exec_sql(`
    CREATE TABLE IF NOT EXISTS autopilot_messages ( ... );
    CREATE INDEX IF NOT EXISTS idx_autopilot_messages_unack ON ...;
  `);
}
```

### Integration with autopilot loop

In `runAutopilotLoop`, after building the state snapshot (step 1 — OBSERVE), inject unacknowledged messages into the prompt:

```typescript
// Between OBSERVE and DECIDE steps:
const pendingMessages = getUnacknowledgedMessages(blueprintId);
if (pendingMessages.length > 0) {
  // Include in prompt as a "User Messages" section
  // Auto-acknowledge after injecting (autopilot has now "seen" them)
  for (const msg of pendingMessages) {
    acknowledgeMessage(msg.id);
  }
}
```

This follows the same pattern as how `buildStateSnapshot` handles insights (marks them read after the autopilot "sees" them at `autopilot.ts:575-578`).

## Section 2 — Lightweight Read-Only Endpoints

These endpoints let the autopilot self-serve context instead of relying on the monolithic `buildStateSnapshot` to pack everything into every prompt.

### `GET /api/blueprints/:id/nodes/summary`

Returns a compact array of node metadata — no descriptions, no handoffs.

**Response:**
```typescript
Array<{
  id: string;
  seq: number;
  title: string;
  status: MacroNodeStatus;
  roles: string[];
  dependencies: string[];
}>
```

**Implementation:** Simple query against `macro_nodes` table, filtering by `blueprint_id`, ordering by `"order"`. Resolves `dependencies` and `roles` from JSON columns. No joins needed.

**File:** `plan-routes.ts` — add after the existing `GET /api/blueprints/:id` handler (~line 200).

### `GET /api/blueprints/:id/nodes/:nodeId/context`

Returns full node details for a specific node, plus resolved dependency titles and latest handoff artifact.

**Response:**
```typescript
{
  id: string;
  seq: number;
  title: string;
  description: string | null;
  prompt: string | null;
  status: MacroNodeStatus;
  error: string | null;
  roles: string[];
  dependencies: Array<{ id: string; seq: number; title: string; status: MacroNodeStatus }>;
  latestHandoff: string | null;  // content from most recent handoff_summary artifact
}
```

**Implementation:**
1. Fetch node from `getBlueprint(id).nodes.find(n => n.id === nodeId)`.
2. Resolve dependencies by mapping `node.dependencies` to the full node list.
3. Fetch latest handoff via `getArtifactsForNode(nodeId)`, filter by `type === "handoff_summary"`, take last.

**File:** `plan-routes.ts` — add near the existing node-level GET endpoints (~line 348).

### `GET /api/blueprints/:id/progress`

Returns aggregate progress counts only — minimal payload for quick status checks.

**Response:**
```typescript
{
  total: number;
  done: number;
  failed: number;
  pending: number;
  skipped: number;
  running: number;
  queued: number;
}
```

**Implementation:** Single pass over `blueprint.nodes` with a status counter. Follows the same counting logic already in `buildStateSnapshot` (`autopilot.ts:582-591`).

**File:** `plan-routes.ts` — add after the nodes/summary endpoint.

### Affected modules

| Module | Change |
|---|---|
| `plan-routes.ts` | 3 new GET handlers |
| `frontend/src/lib/api.ts` | 3 new fetch functions (for frontend use / future autopilot REST calls) |

### Performance considerations

These endpoints are intentionally simple — no sub-agent calls, no Claude invocations. They are synchronous SQLite reads, returning in <5ms. The autopilot can call them via the new tool wrappers (Section 3).

## Section 3 — Autopilot Tool Palette Changes

### New read tools

| Tool | Description | Backend call |
|---|---|---|
| `get_node_titles()` | Returns compact node summary (id, seq, title, status, roles, deps) | `GET /nodes/summary` |
| `get_node_details(nodeId)` | Returns full node with description, prompt, error, resolved deps, latest handoff | `GET /nodes/:nodeId/context` |
| `get_node_handoff(nodeId)` | Returns only the latest handoff artifact content for a node | Inline: `getArtifactsForNode(nodeId)` filtered |

### New message tools

| Tool | Description | Backend call |
|---|---|---|
| `read_user_messages()` | Returns unacknowledged user messages | `getUnacknowledgedMessages(blueprintId)` |
| `acknowledge_message(messageId)` | Marks a message as read | `acknowledgeMessage(messageId)` |

### Removed sub-agent tools

These tools currently spawn one-shot Claude sub-sessions (`enrichNodeInternal`, `splitNodeInternal`, etc.). The refactored autopilot replaces them with direct CRUD:

| Removed tool | Replacement workflow |
|---|---|
| `enrich_node(nodeId)` | `get_node_details(nodeId)` then think about better title/description then `update_node(nodeId, {title, description})` |
| `split_node(nodeId)` | `get_node_details(nodeId)` then decide decomposition then `batch_create_nodes([...])` + `skip_node(nodeId, "split into sub-nodes")` |
| `smart_dependencies(nodeId)` | `get_node_titles()` then analyze dependency graph then `update_node(nodeId, {dependencies: [...]})` — *Note: requires adding `dependencies` to `update_node` params* |
| `reevaluate_node(nodeId)` | `evaluate_node(nodeId)` already exists and is kept |
| `reevaluate_all()` | Autopilot can iterate: `get_node_titles()` then for each non-done node call `evaluate_node(nodeId)` |

### Updated `TOOL_DESCRIPTIONS`

```
### Context (NEW)
- **get_node_titles()** — Get a compact summary of all nodes (id, seq, title, status, roles, dependencies). Use to understand the full blueprint structure without loading details.
- **get_node_details(nodeId)** — Get full details for a specific node: description, prompt, error, resolved dependency titles, and latest handoff artifact content.
- **get_node_handoff(nodeId)** — Get only the latest handoff artifact content for a node. Useful for understanding what a completed node produced.

### User Messages (NEW)
- **read_user_messages()** — Read unacknowledged messages from the user. Check this regularly to stay responsive to user steering.
- **acknowledge_message(messageId)** — Mark a user message as read/handled.

### Node Execution
- **run_node(nodeId)** — Execute a single pending/queued node. Dependencies must be done first.
- **resume_node(nodeId, feedback?)** — Resume a node's session with optional guidance/feedback string.
- **evaluate_node(nodeId)** — Trigger evaluation on a completed node to check quality.

### Node CRUD
- **create_node(title, description, dependsOn?, roles?)** — Create a new node.
- **update_node(nodeId, {title?, description?, prompt?, dependencies?})** — Modify a node. Use for enrichment (improve title/description) and dependency management.
- **skip_node(nodeId, reason)** — Skip a node that is no longer needed.
- **batch_create_nodes([{title, description, dependsOn?, roles?}])** — Create multiple nodes at once. Use for splitting a complex node into sub-nodes.
- **reorder_nodes([{id, order}])** — Change execution order.

[... remaining sections unchanged ...]
```

### `update_node` expansion

Currently `update_node` in `executeDecision` only handles `title`, `description`, `prompt`. It needs to also accept `dependencies` to replace `smart_dependencies`:

```typescript
case "update_node": {
  const nodeId = p.nodeId as string;
  const patch: Record<string, unknown> = {};
  if (p.title) patch.title = p.title;
  if (p.description) patch.description = p.description;
  if (p.prompt) patch.prompt = p.prompt;
  if (p.dependencies) patch.dependencies = p.dependencies;  // NEW
  const updated = updateMacroNode(blueprintId, nodeId, patch);
  // ...
}
```

### Implementation in `executeDecision`

The new read tools do not need HTTP calls — they call the same DB functions directly:

```typescript
case "get_node_titles": {
  const blueprint = getBlueprint(blueprintId);
  if (!blueprint) return { success: false, message: "Blueprint not found", error: "not_found" };
  const summary = blueprint.nodes.map(n => ({
    id: n.id, seq: n.seq, title: n.title, status: n.status,
    roles: n.roles ?? [], dependencies: n.dependencies,
  }));
  return { success: true, message: JSON.stringify(summary) };
}

case "get_node_details": {
  const nodeId = p.nodeId as string;
  const blueprint = getBlueprint(blueprintId);
  if (!blueprint) return { success: false, message: "Blueprint not found", error: "not_found" };
  const node = blueprint.nodes.find(n => n.id === nodeId);
  if (!node) return { success: false, message: "Node not found", error: "not_found" };
  const depDetails = node.dependencies.map(depId => {
    const dep = blueprint.nodes.find(n => n.id === depId);
    return dep ? { id: dep.id, seq: dep.seq, title: dep.title, status: dep.status } : null;
  }).filter(Boolean);
  const artifacts = getArtifactsForNode(nodeId);
  const handoff = artifacts.filter(a => a.type === "handoff_summary").pop();
  return {
    success: true,
    message: JSON.stringify({
      id: node.id, seq: node.seq, title: node.title,
      description: node.description, prompt: node.prompt,
      status: node.status, error: node.error,
      roles: node.roles ?? [], dependencies: depDetails,
      latestHandoff: handoff?.content ?? null,
    }),
  };
}

case "get_node_handoff": {
  const nodeId = p.nodeId as string;
  const artifacts = getArtifactsForNode(nodeId);
  const handoff = artifacts.filter(a => a.type === "handoff_summary").pop();
  if (!handoff) return { success: false, message: "No handoff artifact found", error: "not_found" };
  return { success: true, message: handoff.content };
}

case "read_user_messages": {
  const messages = getUnacknowledgedMessages(blueprintId);
  return { success: true, message: JSON.stringify(messages) };
}

case "acknowledge_message": {
  const messageId = p.messageId as string;
  const result = acknowledgeMessage(messageId);
  if (!result) return { success: false, message: "Message not found", error: "not_found" };
  return { success: true, message: `Acknowledged message ${messageId}` };
}
```

### Impact on `buildStateSnapshot`

With read tools available, `buildStateSnapshot` can become **much leaner** — it only needs to provide:
- Blueprint metadata (id, title, status, enabledRoles)
- Summary stats (done/failed/pending counts)
- Unread insight count (not full insight bodies)
- Unseen suggestion count (not individual suggestions)
- Pending user message count

The autopilot then uses `get_node_titles()`, `get_node_details()`, and `read_user_messages()` to pull specific context as needed. This dramatically reduces per-iteration token usage for large blueprints.

**Before:** ~2000-5000 tokens per state snapshot (all nodes + suggestions + insights inline)
**After:** ~200-400 tokens per state snapshot + selective reads as needed

| Module | Change |
|---|---|

| `plan-db.ts` | Add `AutopilotMessage` interface + 4 CRUD functions, update `initPlanTables` |

## Section 4 — Route AI Operations Through Autopilot

When the blueprint is in autopilot or FSD mode, manual AI operation requests from the frontend should become **messages** to the autopilot rather than spawning independent sub-agents.

### Pattern

```
Frontend button click
  -> POST /api/blueprints/:id/enrich-node  (existing endpoint)
    -> if executionMode is "autopilot" | "fsd":
        createAutopilotMessage(id, "user", "[enrich] Node {nodeId}: {title}")
        trigger autopilot loop if not already running
        return { status: "queued-as-message" }
    -> else (manual mode):
        existing behavior (enrichNodeInternal)
```

### Affected endpoints

| Endpoint | Message format |
|---|---|
| `POST /blueprints/:id/generate` | `"[generate] {description}"` |
| `POST /blueprints/:blueprintId/enrich-node` | `"[enrich] Node #{seq}: {title}"` |
| `POST /blueprints/:blueprintId/nodes/:nodeId/split` | `"[split] Node #{seq}: {title}"` |
| `POST /blueprints/:blueprintId/nodes/:nodeId/reevaluate` | `"[reevaluate] Node #{seq}: {title}"` |
| `POST /blueprints/:blueprintId/nodes/:nodeId/smart-dependencies` | `"[smart-deps] Node #{seq}: {title}"` |
| `POST /blueprints/:id/reevaluate-all` | `"[reevaluate-all] Re-evaluate all completed nodes"` |

### New endpoint for direct messaging

```
POST /api/blueprints/:id/messages
Body: { content: string }
Response: { id: string; status: "queued" }
```

Creates a `role: "user"` message and triggers the autopilot loop if in autopilot/FSD mode.

### Trigger logic

When creating a message for an autopilot/FSD blueprint, check if the loop is already running (by checking `workspacePendingTasks` for an `"autopilot"` type task). If not running:
- If status is `"running"` or `"approved"`: enqueue `runAutopilotLoop`
- If status is `"paused"`: create message but don't auto-start (user must explicitly resume)

This reuses the existing `enqueueBlueprintTask` pattern from `plan-routes.ts:292-296`.

### Autopilot prompt injection

The `buildAutopilotPrompt` function gets a new section when pending messages exist:

```
## Pending User Messages
The user has sent you the following messages. Address them before continuing normal execution.

1. [2026-03-06T10:15:00Z] "[enrich] Node #3: Implement auth middleware"
2. [2026-03-06T10:16:00Z] "Focus on backend nodes first, skip frontend for now"

After handling each message, acknowledge it with acknowledge_message(messageId).
```

### Manual mode fallback

When `executionMode === "manual"`, all endpoints keep their existing behavior — direct sub-agent invocation via `plan-operations.ts`. No message queue interaction.

### Affected modules

| Module | Change |
|---|---|
| `plan-routes.ts` | Add message-redirect logic to 6 existing POST handlers, add new `POST /messages` endpoint |
| `plan-db.ts` | Already covered in Section 1 |
| `autopilot.ts` | `buildAutopilotPrompt` — inject pending messages section |
| `plan-operations.ts` | No changes (manual mode still calls these directly) |

### Data sources

The chat view interleaves two data sources:

1. **User messages** — from `GET /api/blueprints/:id/messages?limit=50` (new endpoint serving `autopilot_messages` where `role = "user"`)
2. **Autopilot log entries** — from existing `GET /api/blueprints/:id/autopilot-log`

Both are sorted by `createdAt` and merged client-side into a single timeline.

### Interaction flow

1. User types message in chat input, hits Send -> `POST /api/blueprints/:id/messages` -> message appears immediately (optimistic update)
2. If in autopilot/FSD mode: autopilot loop picks up message, acts on it, logs results -> log entries appear via polling
3. If in manual mode: message is stored but no autopilot action. User can still use the existing buttons. The chat serves as a log/history.

### Existing generator textarea migration

The current "Generate nodes" textarea (`generateInstruction` state + button in `page.tsx`) is replaced by the chat input. When the user types in the chat and the blueprint has no nodes, the message is routed as a generate request. The existing `generatePlan` API function is replaced by `POST /messages` with content like `"Generate: {user's instruction}"`.

### Phase 1: Message Queue + Read Endpoints (backward-compatible)

**Scope:** `plan-db.ts`, `plan-routes.ts`, `frontend/src/lib/api.ts`

- Add CRUD helpers
- Add 3 read-only endpoints (nodes/summary, nodes/:nodeId/context, progress)

### Phase 2: Read Tools in Autopilot Palette

**Scope:** `autopilot.ts`

- Add `get_node_titles`, `get_node_details`, `get_node_handoff` to `ALL_TOOL_NAMES` and `TOOL_DESCRIPTIONS`
- Add `read_user_messages`, `acknowledge_message` to palette
- Add handlers in `executeDecision`

**Risk:** Low. The autopilot gains new capabilities but existing tools still work. The AI may prefer the new tools organically.

**Verification:** Run autopilot on a test blueprint. Confirm read tools work correctly. Check autopilot log for tool usage.

### Phase 3: Route AI Ops Through Messages (autopilot/FSD mode only)

**Scope:** `plan-routes.ts`, `autopilot.ts`

- Modify 6 POST handlers to redirect to message queue when `executionMode` is autopilot/FSD

- Remove sub-agent tools (`enrich_node`, `split_node`, `smart_dependencies`, `reevaluate_node`, `reevaluate_all`) from palette
- Expand `update_node` to accept `dependencies`
- Slim down `buildStateSnapshot`

**Risk:** Medium. This changes autopilot behavior. Sub-agent removal means the autopilot must handle enrichment/splitting inline. The AI prompt must be updated to guide it through these workflows.

**Mitigation:**
- Manual mode keeps existing behavior (no regression for non-autopilot users)
- Test with both small (5 nodes) and large (30+ nodes) blueprints
- Monitor token usage before/after state snapshot slimming

**Verification:**
- Backend unit tests for message routing logic
- Integration test: create blueprint in autopilot mode, trigger enrich via API, confirm message created and autopilot picks it up
- Manual test of manual-mode fallback

### Phase 4: Frontend Chat UI

**Scope:** `frontend/src/app/blueprints/[id]/page.tsx`, new `AutopilotChat.tsx` component

- Build chat component
- Replace generator textarea
- Wire up polling for messages + autopilot log
- Optimistic updates for sent messages

**Risk:** Low — frontend-only changes. Backend already supports all needed endpoints from Phase 1.

**Verification:** Manual testing of chat flow. Confirm messages appear in real-time when autopilot is running.

## Trade-off Analysis

### Performance

- **Token usage:** Major improvement. Current `buildStateSnapshot` serializes everything (~2-5K tokens). Slimmed snapshot + on-demand reads = ~200-400 tokens base + selective reads. Net savings for large blueprints: 60-80%.
- **Latency:** New read endpoints are synchronous SQLite reads (<5ms). No performance regression.
- **Message queue overhead:** Negligible — one extra table scan per autopilot iteration for unacknowledged messages.

### Maintainability

- **Reduced code duplication:** Removing 5 sub-agent tools from `executeDecision` (~80 lines) and their corresponding `plan-operations.ts` functions (~300 lines, though these stay for manual mode). Net reduction in autopilot-specific code.
- **Single communication channel:** All user-to-autopilot interaction goes through the message queue, making it easier to debug and audit.
- **Clearer separation:** `plan-operations.ts` becomes "manual mode only." `autopilot.ts` becomes self-contained.

### Complexity

- **Prompt engineering:** The autopilot prompt must guide the AI through enrichment/splitting workflows that were previously handled by specialized sub-agent prompts. This shifts complexity from code to prompt engineering.
- **Chat UI:** New frontend component, but relatively straightforward (message list + input).
- **Dual-path logic in routes:** 6 existing endpoints gain an `if (isAutopilot) { createMessage } else { existingBehavior }` branch. This is simple but adds conditional paths.

### Backward compatibility

- **Manual mode:** Fully preserved. All existing behavior unchanged.

## Appendix: Interaction Sequence (Autopilot Mode)

```
User types "Enrich node #3" in chat
  -> POST /api/blueprints/:id/messages { content: "Enrich node #3" }
    -> createAutopilotMessage(id, "user", "Enrich node #3")
    -> autopilot loop running? -> yes -> message will be picked up next iteration
                                -> no  -> enqueueBlueprintTask -> runAutopilotLoop

Autopilot iteration N:
  1. OBSERVE: buildStateSnapshot (slim) + getUnacknowledgedMessages
  2. Prompt includes: "User message: Enrich node #3"
  3. DECIDE: AI returns { action: "get_node_details", params: { nodeId: "abc" } }
  4. EXECUTE: returns node details
  5. LOG

Autopilot iteration N+1:
  1. OBSERVE: state + no new messages (previous was acknowledged)
  2. AI has node details in context from previous iteration
  3. DECIDE: AI returns { action: "update_node", params: { nodeId: "abc", title: "...", description: "..." } }
  4. EXECUTE: node updated
  5. DECIDE: AI returns { action: "acknowledge_message", params: { messageId: "xyz" } }
  6. Continue normal execution...
```

Note: The multi-iteration workflow (read then decide then update) means enrichment takes 2-3 autopilot iterations instead of 1 sub-agent call. This is acceptable because:
1. Autopilot iterations are fast (~2-5s each for the decision call)
2. The autopilot has full context of the blueprint and can make better enrichment decisions than a one-shot sub-agent
3. No additional Claude session startup overhead (the autopilot is already running)
